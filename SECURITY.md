# 安全策略

## 支持范围

安全修复面向最新 1.x 稳定版本。报告问题时请提供言韧版本、言序版本、最小复现、策略参数、执行入口和预期安全边界；不要附带真实凭据、生产请求正文或用户数据。敏感问题应使用 GitHub 仓库的私密安全报告渠道，不要先公开利用细节。

## 权限边界

言韧不申请文件、网络、监听、环境、进程或原生扩展权限。它会同步调用应用传入的操作、重试判定、等待器、观察器、停止判定和时间源；这些回调拥有顶层应用授予的能力。不要把不可信脚本直接作为回调。

## 重试放大

重试会放大请求、写入、日志和下游负载。应用必须：

- 只重试明确的暂时性错误，优先使用`可重试代码`白名单；
- 为写操作使用幂等键、唯一约束、事务或业务去重；
- 设置符合下游容量的次数、延迟和断路阈值；
- 把调用方截止时间接入`设停止判定`；
- 监控`等待`、`耗尽`、`停止`事件和断路器拒绝计数；
- 避免在多层客户端、代理和服务中叠加未协调的重试。

`总是重试`只适合已证明幂等且错误域受控的操作或测试，不是生产默认策略。

## 资源拒绝服务

尝试次数、单次延迟、累计计划等待、倍率、错误代码数量、失败阈值和恢复时间都有硬上限。完整计划按尝试数线性生成，退避在乘法前封顶。上限保护库自身，但业务操作和回调仍可能消耗任意资源；顶层应用还应设置请求体、连接、并发、队列和总截止时间限制。

## 回调失败

判定、等待、观察和停止回调失败分别转换为`YANREN_PREDICATE`、`YANREN_WAITER`、`YANREN_OBSERVER`和`YANREN_STOP_CALLBACK`。这些错误不会作为业务错误继续重试。观察器应快速、无阻塞并自行限制缓冲；不要在观察器中递归调用同一重试流程。

## 错误数据

`错误详情`保留源错误代码、类别、位置和踪迹。这些字段可能泄露路径、实现细节或下游消息。对外响应和遥测必须按数据策略筛选；不要直接把完整详情返回给不可信客户端。`可重试代码`只读取约定代码，不会清洗错误消息。

## 时间与断路器

断路器时间源必须返回非负、安全、单调不减的毫秒值。倒退会立即失败，避免冷却被绕过或无限延长。自定义时间源和等待器应使用同一时间域；生产中优先使用单调时钟适配器。断路器是进程内保护，不是认证、限流、配额或分布式一致性机制。
