# 言韧使用指南

## 重试语义

`重试策略(最大次数, 初始延迟秒, 倍率, 最大延迟秒)`描述一次执行的完整计划。`最大次数`包含首次调用，因此最大等待次数是`最大次数 - 1`。操作接收从 1 开始的次数；判定器只在操作失败、且尚有下一次机会时调用。

```yanxu
定 策略 为 言韧.重试策略（5，0.2，2，2）；
```

无抖动时，第`n`次失败后的基础延迟为：

```text
min(初始延迟秒 × 倍率^(n - 1), 最大延迟秒)
```

实现使用逐步封顶乘法，不直接计算可能溢出的幂。`基础延迟秒`返回无抖动值，`延迟秒`返回实际值，`延迟表`返回所有等待，`计划表`还给出失败次数和累计延迟。

`立即策略(次数)`等价于初始延迟、最大延迟都为零的常量策略。它不决定是否真实等待；`立即执行`和`立即执行控制`才替换等待行为。

## 可复现抖动

`设抖动(比例, 种子)`启用向下抖动：

```text
实际延迟 = 基础延迟 × ((1 - 比例) + 比例 × 随机小数(种子 + 失败次数))
```

比例为 0 时不抖动；比例为 1 时得到`[0, 基础延迟)`的完全抖动。同一策略、种子和失败次数在树解释器与字节码 VM 中产生相同值。种子是配置，不是秘密，也不提供密码学随机性。

```yanxu
定 策略 为 言韧.默认策略（）.设抖动（1，2026）；
定 表 为 言韧.延迟表（策略）；
```

为避免多个客户端同步重试，每个逻辑操作或客户端实例应选择不同种子。测试应固定种子。

## 选择可重试错误

`总是重试`和`从不重试`是边界工具。生产代码优先使用稳定错误代码白名单：

```yanxu
定 可恢复：法 为 言韧.可重试代码（
    【「YANQIU_TIMEOUT」，「YANQIU_CONNECT」，「YANKU_BUSY」】
）；
```

错误消息若以`YAN`开头、后接大写字母、数字或下划线，并使用中文冒号分隔，`错误详情`会恢复代码与消息。白名单在创建时复制，调用方随后修改原列不会改变判定器。

判定器签名是`法(所误:误, 次数:数):理`。它不应执行网络请求、真实等待或修改业务状态。判定器抛错会变为`YANREN_PREDICATE`，不会继续重试。

## 执行入口

### 传播结果或错误

- `执行`使用真实等待；
- `立即执行`执行同一计划但跳过等待；
- `受控执行`使用调用方提供的`执行控制`；
- 对应的“经断路器”入口在每次尝试外包裹同一个断路器。

成功时直接返回业务值。业务操作最终失败、判定拒绝或次数耗尽时，原始业务`误`原样抛出。停止、回调和内部状态错误使用`YANREN_`代码。

### 结构化结果

`尝试执行`和`受控尝试执行`不抛最终错误，而返回固定形状：

```text
成功：{成功: 真,  值: <业务值>, 错误: 空,       次数: <实际调用数>}
失败：{成功: 假,  值: 空,       错误: <错误详情>, 次数: <实际调用数>}
```

`真实等待`参数只在兼容入口`尝试执行`中选择默认控制或立即控制。新代码需要观察、停止或自定义调度时使用`受控尝试执行`。

## 执行控制

`执行控制()`默认使用系统等待、忽略事件且永不停止。三个构建器可组合：

```yanxu
定 控制 为 言韧.执行控制（）
    .设等待器（等待器）
    .设观察器（观察器）
    .设停止判定（停止判定）；
```

回调契约：

| 回调 | 签名 | 调用时机 |
| --- | --- | --- |
| 等待器 | `法(秒:数, 上下文:典):空` | 已决定重试并发出`等待`事件后 |
| 观察器 | `法(事件:典):空` | 每个可观察阶段 |
| 停止判定 | `法(上下文:典):理` | 首次和每次后续尝试之前 |

停止判定收到阶段为`检查`的上下文，其中`次数`是下一次尝试编号。返回真时，观察器收到`停止`事件，然后执行以`YANREN_STOPPED`结束。它是协作式停止：正在执行的业务操作不会被抢占。

等待器收到的`累计等待秒`已经包含本次计划延迟。等待器失败时，该数值表示已调度的逻辑等待，不证明真实时间已经流逝。

观察器在业务操作外调用。即使`成功`事件的观察器失败，已经成功的业务操作也不会被再次执行。

## 生命周期事件

事件模式版本为 1，固定字段为：

```text
模式版本、阶段、次数、最大次数、累计等待秒、等待秒、错误
```

典型成功序列：

```text
开始 → 失败 → 等待 → 开始 → 成功
```

停止判定使用的`检查`上下文不发送给观察器。各阶段精确定义、指标建议和字段含义见[可观察性契约](OBSERVABILITY.md)。

## 断路器

`断路器(失败阈值, 恢复毫秒)`按连续失败工作：

1. 关闭状态允许调用；失败达到阈值后开启。
2. 开启状态拒绝调用，不因额外失败重置冷却起点。
3. 冷却到期后转为半开。
4. 半开只允许一个已占用探针；成功关闭，失败重新开启并重启冷却。

`是否允许()`只查看，不占用许可。手工流程必须使用：

```yanxu
若 所断路器.获取许可（） 则
    试 则
        定 值 为 操作（）；
        所断路器.记成功（）；
    救 所误 则
        所断路器.记失败（）；
        抛 所误；
    终
终
```

推荐直接使用`调用(操作)`，它保证成功与失败都完成状态记录。被拒绝的操作不会执行，并抛`YANREN_OPEN`。

`重置()`关闭断路器但不清空累计成功、失败、拒绝和开启计数。`记成功()`会增加成功总数并关闭；它可用于确认外部恢复。`快照()`可能在读取时把已到期的开启状态推进到半开。

## 确定性时间

断路器默认使用系统毫秒。关闭状态下可替换为无参数时间源：

```yanxu
令 当前毫秒：数 为 0；

法 虚拟时间（）：数 则
    归 当前毫秒；
终

定 所断路器 为 言韧.断路器（2，1000）.设时间源（虚拟时间）；
```

时间源在安装时立即校验，之后必须单调不减。自定义等待器应推进同一时间变量，使重试等待与断路冷却处于同一时间域。开启或半开状态不能更换时间源。

## 组合重试与断路器

```yanxu
定 结果 为 言韧.受控执行经断路器（
    策略，所断路器，业务操作，可恢复，控制
）；
```

每次业务尝试都通过断路器。断路器拒绝产生`YANREN_OPEN`；只有判定器允许时，重试层才会继续等待并再次尝试。使用`总是重试`时要确保等待计划能覆盖断路冷却，并受调用方总截止时间约束。

## 常见陷阱

- 把最大次数理解成“额外重试次数”；它实际包含首次调用。
- 对非幂等写入使用重试，造成重复扣款、消息或订单。
- 多层都重试，导致乘法级请求放大。
- 用`是否允许`后直接执行手工探针；它不占用半开许可。
- 取得许可后遗漏`记成功`或`记失败`，令半开探针长期占用。
- 让观察器阻塞、无限缓存或递归进入同一执行链。
- 用系统墙钟和虚拟等待器混合驱动断路器。
- 把抖动种子当作安全随机数或秘密。
