# 言韧架构

## 依赖方向

言韧没有第三方包依赖，只使用言序的数学、随机、正则和时间标准库。HTTP、请求、数据库、消息队列和任务库可以依赖言韧；言韧通过操作法、错误代码和回调与它们协作，不反向导入上层库，因此不会形成包循环。

## 分层

实现分为四层：

1. 策略层验证次数、延迟、倍率、累计预算和抖动种子。
2. 计划层计算基础延迟、抖动延迟、延迟表和结构化计划表。
3. 执行层负责尝试、错误判定、事件、停止和等待器隔离。
4. 断路层维护关闭/开启/半开状态、许可、时间和计数。

组合入口只把每次业务操作包裹进同一个断路器，再交给执行层；断路层不知道重试策略，执行层也不直接修改断路器状态。

## 退避算法

策略构造以单次线性扫描核算累计等待。完整延迟表和计划表维护“当前基础延迟”，每一项只推进一次，因此为`O(最大次数)`。单点`基础延迟秒`从初始值推进到目标或封顶点；零延迟、倍率 1 和已在上限的策略直接返回。

乘法前先比较`当前 > 最大延迟 / 倍率`，再决定封顶，避免先生成无穷大或不可用数。抖动只向下缩短基础延迟，因此不会突破策略构造时验证的累计预算。

## 执行状态

每轮执行按以下顺序：

1. 建立下一次`检查`上下文并调用停止判定；
2. 增加尝试次数并发送`开始`；
3. 在隔离捕获边界内调用业务操作；
4. 成功则发送`成功`并返回；
5. 失败则保存结构化错误并发送`失败`；
6. 若耗尽则发送`耗尽`并抛原错误；
7. 调用判定器，拒绝时发送`放弃`并抛原错误；
8. 计算等待，发送`等待`，再调用等待器。

业务操作是唯一会被重试的区域。停止、观察、判定和等待回调在区域外调用，各自失败时转换为独立库错误。这样观察成功失败不会重放已经完成的业务副作用。

## 错误模型

库错误统一使用`YANREN_<代码>：消息`。`错误详情`识别组织约定的`YAN[A-Z0-9_]+：`前缀，因此判定器既能处理本库错误，也能处理请求、数据库等上层库的稳定代码。无法识别的错误归为`YANREN_RUNTIME`，但直接执行入口仍原样抛出业务错误。

## 断路状态机

断路器把状态、连续失败、开启时刻、半开占用和累计计数保存在实例中。状态读取只在开启时访问时间源；冷却到期后原地转为半开。`获取许可`是唯一会占用半开探针的公开入口，`是否允许`保持只读查询语义。

`调用`先取得许可，再在隔离边界内执行无参业务法。状态记录发生在捕获边界外，因此计数溢出、时间源或状态维护错误不会被误记为新的业务失败。

开启期间的额外`记失败`增加失败总数，但冷却尚未到期时不移动开启时刻。半开失败会再次开启并增加开启总数。

## 时间模型

时间源返回非负安全整数毫秒。实例保存上次读取值并拒绝倒退。安装自定义时间源时立即读取一次，以便配置阶段尽早失败。库不在后台轮询；状态只在调用`状态`、许可、失败记录、剩余时间或快照时推进。

## 所有权与并发

错误代码白名单在构造判定器时复制。事件和快照每次新建；调用方修改它们不会回写执行或断路状态。策略和执行控制是可配置对象，配置完成后应按只读方式共享。

实现面向言序同步执行流，不包含锁、线程或跨进程协调。需要并发共享断路器时由应用串行化访问；需要集群级状态时使用外部控制面。
