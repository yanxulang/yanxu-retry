# 性能与资源预算

## 硬上限

| 项目 | 上限 |
| --- | ---: |
| 最大尝试次数 | 4096 |
| 单次延迟 | 86400 秒 |
| 计划累计等待 | 604800 秒 |
| 退避倍率 | 1000 |
| 错误代码白名单 | 64 项 |
| 断路失败阈值 | 4096 |
| 断路恢复时间 | 604800000 毫秒 |
| 安全整数绝对值 | 9007199254740991 |

这些是拒绝失控配置的库级硬边界，不是推荐默认值。面向用户请求的策略通常应远小于它们，并受调用方总截止时间约束。

## 复杂度

- 策略构造核算完整无抖动预算，为`O(最大次数)`；零延迟、倍率 1 或初始即封顶时为常数时间。
- 单点基础延迟为`O(min(失败次数, 到达封顶所需次数))`。
- `延迟表`和`计划表`维护递推值，为`O(最大次数)`时间和输出空间。
- 抖动每项调用一次显式种子随机函数，为常数额外成本。
- 一次执行的库调度为`O(实际尝试数)`，不含业务操作、回调和真实等待。
- 代码白名单判定最坏扫描 64 项。
- 断路器状态、许可、记录和快照均为常数时间。

计划构造先比较除法封顶条件，再乘法；不会为超大指数生成无穷数。抖动只缩短基础值，因此总等待不超过构造时验证的无抖动预算。

## 回归工作负载

`benchmarks/退避与状态机.yx`在言序默认一百万执行步预算内完成：

- 16 份 15 项抖动计划，共 240 个计划项；
- 64 轮第四次成功的立即执行，共 256 次业务调用；
- 128 组断路失败/成功状态更新与累计计数验证。

发布门禁在言序 1.1.6 和当前稳定版中分别运行树解释器与字节码 VM，并比较最终值、输出和错误。该文卷是确定性回归工作负载，不是跨机器吞吐基准。

## 调优建议

1. 先缩小最大次数，再调整延迟；大量零延迟尝试仍会消耗 CPU 和下游容量。
2. 对同一依赖的客户端使用不同抖动种子，降低同步突发。
3. 观察器只做常数时间入队或计数，复杂编码与 I/O 放到外部消费者。
4. 停止判定读取已计算的取消或截止状态，不在回调中阻塞。
5. 只在诊断、展示或测试时生成完整计划表；执行路径按需计算每次延迟。
6. 让断路器阈值和恢复时间与重试计划协同，避免所有重试都落在开启窗口。
7. 在真实业务形状、下游延迟和并发下做容量测试；不要把此回归文卷的运行时间当作服务指标。
